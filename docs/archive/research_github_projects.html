Top 5 unimplemented ideas

Ranked by debug value, fit with thin-client model, and reuse of existing wiring.

1 Outbox row inspector
Inspired by: bull-board flower pgweb (scoped subset)

Gap Schema drawer shows outbox counts only. When the alert bar fires, there is no in-UI path to the failing row's error message or entity type.

Fit Outbox health is the #1 critical alert. outbox.py already queries neo4j_outbox and respects SM_IGNORED_OUTBOX_IDS. A read-only API with whitelisted columns keeps the no-secret model.

Proposed API

GET /api/outbox?status=failed&limit=50
→ { rows: [{ id, status, entity_type, error, created_at, ignored }] }

2 CLI event stream + MCP for AI agents
Inspired by: spotlight (spotlight tail, spotlight mcp)

Gap CLI is loop / serve / check only. Agents using memory_bridge.py cannot ask “is REM backlog growing?” without opening the browser.

Fit Shared Memory is a multi-agent workstation surface. Spotlight's MCP pattern maps directly: agents save facts → dream cycle processes → agents observe pipeline state headlessly.

Proposed commands

python -m sm_telemetry_monitor tail [--follow] [--format json|logfmt]
python -m sm_telemetry_monitor mcp   # stdio MCP: get_pipeline_status, get_health, list_outbox_failures

3 Prometheus /metrics export
Inspired by: flower (built-in Prometheus + prometheus.yml in repo)

Gap Metrics live in SQLite and dashboard JSON only. No overnight alerting when outbox_failed > 0 or the poll loop stops.

Fit Observe-don't-control philosophy — exporting gauges is safer than Flower-style remote worker control. Homelab users often already run Grafana/Alertmanager.

4 Correlated incident timeline
Inspired by: langfuse (trace → logs/spans) spotlight (unified stream)

Gap Dashboard (/) and logs (/logs) are separate. On a backlog spike, users manually guess which log source and time window to open.

Fit REM/NREM debugging is temporal — a change at T should surface gateway journal + REM audit ±5 min at T. All sources exist in logs_reader.py; correlation is composition, not new gateway instrumentation.

5 Doctor-driven external monitor emit
Inspired by: AutoKuma (declarative JSON/TOML → Uptime Kuma)

Gap doctor.py has a FEATURE_MATRIX and check prints readiness, but emits nothing for external uptime tools. In-browser alerts only help when the tab is open.

Fit Extends portability (SHARED_MEMORY_ROOT, check-env.sh) into unattended ops without building notifications inside the monitor. Generated files contain URLs and paths only — never tokens.

Proposed command

python -m sm_telemetry_monitor check --emit-monitors uptime-kuma.json

Suggested implementation order

  1. Outbox row inspector — highest debug value; smallest scope; uses existing Postgres path.
  2. CLI tail — quick win before MCP; unblocks headless agent workflows.
  3. Prometheus export — unlocks unattended alerting via existing Grafana stack.
  4. Incident timeline — best operator UX; composes dashboard + logs + outbox.
  5. Doctor monitor emit — completes portability story; delegates notifications externally.
